## 2. 当前项目进度判断

结合文档目标与现有代码，当前项目整体处于：

`V2 核心后端框架已基本成型，已完成基础可运行验证，并已落地 Web 最小可用闭环（身份 + 会话 + 历史 + 聊天页面）；但距离正式产品化与实时通话形态仍有明显距离。`

更具体一点，可以拆成下面几层判断。

### 2.1 已经完成或基本完成的部分

#### 1. V2 基础架构已经落地

这一点是当前仓库最明确的成果。

现有代码已经形成比较清楚的分层：

- `domain`：统一消息、上下文、回复结构
- `ports`：存储、记忆、LLM、transport 抽象
- `infra`：QQ/Web/Repo/LLM/Scheduler 等实现
- `application/services`：提醒、备忘录、用户、脚本工作区等业务服务
- `application/commands`：命令系统
- `application/agent`：主 Agent、Skill 注册、子 Agent 技能
- `application/event_handler.py`：统一事件入口

这说明 PDF 计划稿里“先让 V2 活下来、让大家看懂”的第一步，实际上已经做出来了。

#### 2. QQ 主链路已经可用

从以下文件看，QQ 入口已经是项目当前最成熟的链路：

- [`src/luoying_bot/main_qq.py`](../src/luoying_bot/main_qq.py)
- [`src/luoying_bot/bootstrap.py`](../src/luoying_bot/bootstrap.py)
- [`src/luoying_bot/infra/transports/qq_ws_transport.py`](../src/luoying_bot/infra/transports/qq_ws_transport.py)

目前已经具备：

- QQ WebSocket 接入
- 统一消息解析
- 命令分发
- Agent 回复
- 提醒恢复与调度
- 群运行时控制

这意味着项目不是纯设计稿，而是已经有实际可运行主体。

#### 3. Web 最小可用闭环已经落地

这一点和 PDF 中“尽快建立非 QQ 替代调试入口”的目标一致。

现状如下：

- [`src/luoying_bot/main_web.py`](../src/luoying_bot/main_web.py) 提供 FastAPI 应用入口
- [`src/luoying_bot/bootstrap_web.py`](../src/luoying_bot/bootstrap_web.py) 提供 Web 专用容器
- [`src/luoying_bot/infra/web/api.py`](../src/luoying_bot/infra/web/api.py) 提供 `POST /chat`、`/api/chat`、`/api/sessions`、`/api/sessions/{id}/messages`
- [`src/luoying_bot/infra/web/session_store.py`](../src/luoying_bot/infra/web/session_store.py) 提供会话与消息历史存储
- [`src/luoying_bot/infra/web/index.html`](../src/luoying_bot/infra/web/index.html) 提供可直接使用的最小前端页面
- [`src/luoying_bot/infra/transports/web_noop_transport.py`](../src/luoying_bot/infra/transports/web_noop_transport.py) 提供 Web 场景下的最小 transport

结论：

- 项目已经不再完全依赖 QQ 才能测试主逻辑
- Web 入口已经具备“可演示、可使用”的最小闭环，但仍是轻量 MVP

#### 4. Agent + Skill 体系已经进入可扩展状态

当前主 Agent 并不是简单问答，而是已经具备一定工程化能力：

- `SkillRegistry` 自动注册技能
- `AgentService` 使用 ReAct 风格循环进行技能选择
- 已实现提醒、天气、联网搜索、备忘录、群信息、图片分析、编程子 Agent 等能力

这说明 V2 的“可扩展多能力 Agent 系统”目标，已经有真实骨架，不再只是概念。

#### 5. 部分业务能力已经具备持久化与运行机制

当前已经能看到比较完整的业务能力沉淀：

- 用户资料 JSON 持久化
- 提醒任务持久化与恢复
- 备忘录按用户存储
- 脚本工作区隔离
- 会话记忆

这说明后端并不只是聊天 demo，而是在向“长期可维护系统”靠近。

### 2.2 已完成但还不够稳的部分

#### 1. Web 前端已可用，但仍是 MVP 层级

当前 Web 端已经有：

- 会话创建/查询
- 历史消息查询
- 页面内聊天闭环

但仍未完成的产品化能力包括：

- 登录体系
- 鉴权体系
- 流式回答体验
- 前后端联调规范

所以它已从“纯调试入口”升级到“可用 MVP”，但还不是完整产品前端。

#### 2. 架构文档和操作文档已明显改善，但还可以继续收口

目前仓库中已经有：

- 开发计划稿
- 实时通话设计稿
- 后端流程图
- README

但从工程角度看，仍需继续补强：

- 统一的协作边界说明
- 更系统的变更记录与回滚说明
- 更细粒度的故障排查手册

这意味着“能跑”和“团队稳定接手”之间还有一段距离。

#### 3. 代码中仍有明显实验性和 AIGC 痕迹

从注释、代码风格、异常处理方式可以看出：

- 一些模块已经可用
- 但整体还没进入“严格收口、统一规范”的阶段

这类问题短期不一定阻塞功能，但会影响长期维护和协作效率。

### 2.3 还没有真正开始或尚未落地的部分

#### 1. 服务化拆分试点尚未开始

PDF 计划稿中提到优先试点：

- 天气模块服务化
- 论文模块服务化

但当前代码里，天气能力仍是本地 skill，论文服务也未看到独立实现。

说明“主 Agent + 外部服务 + 子 Agent”的演进方向还停留在规划阶段。

#### 2. 正式 Web 前端工程尚未出现（但最小页面已落地）

计划稿里要求的最小 Web 前端应至少包含：

- 登录页
- 对话页
- 历史记录页
- 基础个人中心页

当前仓库中已有内嵌静态页面实现最小闭环，但尚未拆分为独立前端工程，也未进入完整产品化阶段。

#### 3. 实时音视频通话后端已完成前两步，但媒体轨仍未落地

[`documents/live2d_webRTC_backend_design.md`](./live2d_webRTC_backend_design.md) 中规划的完整模块尚未全部落地，但当前代码已经出现实时入口与信令骨架：

- `src/luoying_bot/main_realtime.py`
- `src/luoying_bot/bootstrap_realtime.py`
- `src/luoying_bot/infra/transports/realtime_ws_transport.py`

当前已实现：

- 独立实时服务入口（与主聊天链路解耦）
- 会话创建/查询/关闭接口
- WebSocket 信令通道（offer/answer/ice）
- `to: "server"` 协商链路增强（嵌套 offer、对象 candidate、end-of-candidates）
- `server_signaling_target` 已从 policy 注入 transport（非硬编码）

当前仍未完成：

- 占位视频轨/音频轨
- Live2D 驱动事件细化

同时，[`requirements.txt`](../requirements.txt) 已加入 `aiortc` 依赖，运行环境可直接按依赖安装。

因此可以明确判断：

`实时通话后端已从“纯信令骨架”进入“协商可测”阶段，当前瓶颈转为媒体轨实现与真机联调。`

#### 4. 自动化测试已建立基础回归网，但覆盖范围仍需继续扩大

当前仓库已经有 `tests/` 目录、基础 smoke tests、启动自检与 Ollama 状态检查脚本，并补齐了以下高价值回归：

- `EventHandler`
- `CommandDispatcher`
- `ReminderService`
- `Web API` 关键错误分支（`/chat`、`/api/chat`）

这意味着：

- 核心消息主链路的回归风险已显著下降
- 但 `ScriptWorkspaceService`、Web API 边界、LLM 异常分级等仍需继续补测

## 3. 当前阶段的真实定位

如果用一句话概括当前状态：

`项目已经完成“V2 后端骨架搭建”和“脱离 QQ 的最低限度调试能力建设”，接下来应该进入“可维护化、可产品化、可扩展化”的第二阶段。`

也就是说，当前策略应保持“实时优先 + 主链稳态兜底”：

1. 保持基础回归网持续可用
2. 直接推进实时媒体层（占位轨 + 真机联调）
3. 在不破坏主链/QQ 的前提下回补 Web 稳定性问题
4. 之后再推进服务化试点与更重的产品化项

---

## 4. 后续建议总路线

建议后续按五个阶段推进：

1. 工程收口阶段
2. Web 最小可用闭环阶段
3. 会话与数据能力补全阶段
4. 服务化拆分试点阶段
5. 实时通话与 Live2D 后端阶段

下面按阶段展开。

---

## 5. 分阶段推进建议

### Phase 1：工程收口阶段

### 目标

把当前“能跑”的后端整理到“可稳定继续开发”的状态。

### 为什么先做这个

因为当前主体能力已经不少，但测试、部署说明、配置说明仍然偏薄。如果现在直接继续堆功能，后面会越来越难维护。

### 建议任务

1. 补充部署与启动文档

- 明确服务器部署步骤
- 明确 Python 版本要求
- 明确依赖安装方式
- 明确环境变量说明
- 明确 QQ / Web 两条启动路径

2. 整理配置说明

- 检查 `.env.example`
- 标明哪些配置是必须的
- 标明哪些配置只在 QQ 模式需要
- 标明哪些配置只在 Web / 图片 / 编程子 Agent 下需要

3. 增加最基础的自动化测试

优先补这些最值得先测的模块：

- `EventHandler`
- `CommandDispatcher`
- `ReminderService`
- `ScriptWorkspaceService`
- `UniMessage` 消息转换

4. 整理日志与异常输出

- 减少随意 `print`
- 统一关键模块日志方式
- 明确启动失败与运行失败的可观测信息

### 阶段完成标准

- 新成员按文档能启动项目
- 至少有一组基础测试能跑通
- Web 和 QQ 启动路径说明完整
- 常见配置问题能快速定位

### 这一阶段的产出物

- 部署文档
- 配置说明文档
- 基础测试用例
- 常见问题记录

---

### Phase 2：Web 最小可用闭环阶段

### 目标

把当前 Web 调试入口升级成真正可用于演示和基本使用的独立入口。

### 为什么现在做

当前已经有 `POST /chat`，说明后端最小通路已经具备。继续推进 Web，是最自然、风险也最低的一步。

### 建议任务

1. 明确 Web 侧最小接口集合

建议至少补齐：

- 创建会话
- 发送消息
- 获取历史消息
- 查询当前会话列表

2. 设计最小前端页面

至少包括：

- 登录页或临时身份页
- 对话页
- 历史记录页
- 基础错误提示与加载态

3. 规范 Web 请求与返回模型

- 不要长期只靠一个 `reply: str`
- 为后续流式输出、历史记录、多端扩展预留结构

4. 明确 Web 模式下 transport 与 HTTP 响应的职责边界

当前 `WebNoopTransport` 是合理过渡方案，但后面要避免让 Web 能力长期只靠“特殊分支”维持。

### 阶段完成标准

- 不依赖 QQ 也能完成一次完整对话
- 用户能看到历史消息
- Web 端基本可演示
- 前后端接口开始稳定

### 这一阶段的产出物

- Web 最小前端
- Web 会话接口
- 基础历史记录能力
- 前后端联调说明

---

### Phase 3：会话与数据能力补全阶段

### 目标

把“能聊天”升级成“像一个真正的应用在工作”。

### 为什么单列这一阶段

当前系统有记忆，但更偏运行时；Web 端要真正独立使用，就需要稳定的会话、历史、用户侧数据管理能力。

### 建议任务

1. 明确会话模型

建议引入清晰概念：

- 用户
- 会话
- 消息
- 历史记录

2. 增加历史消息存储

至少让 Web 端能够：

- 查看某会话历史
- 恢复最近对话
- 区分不同会话线程

3. 规范 memory 与 history 的边界

当前 `ConversationMemory` 更像 prompt memory。

后续应区分：

- 给模型看的上下文记忆
- 给用户看的持久化历史记录

4. 为后续流式输出做接口预留

哪怕先不实现流式，也应在协议层避免把自己锁死在“单次纯文本返回”上。

### 阶段完成标准

- Web 用户能管理至少多个会话
- 历史消息能被查询与恢复
- 记忆系统和历史系统职责清晰

### 这一阶段的产出物

- 会话/消息模型设计
- 历史记录仓储
- 历史接口与页面

---

### Phase 4：服务化拆分试点阶段

### 目标

验证 V2 是否真的具备从“单体 Agent”向“主 Agent + 外部服务 + 子 Agent”演进的可行性。

### 建议优先顺序

按原计划稿，优先从：

1. 天气模块
2. 论文模块

开始。

### 为什么不建议先拆更核心的东西

因为这两个模块边界相对清晰、风险较低，适合作为试点。如果一开始就拆主链路，会把整个项目拖进重构泥潭。

### 建议任务

1. 先定义服务接口，再写实现

例如先明确：

- 输入参数
- 输出结构
- 超时策略
- 失败兜底

2. 主 Agent 通过 skill 调服务

不要直接把现有主链路重写成分布式结构，而是先以 skill 的方式接入外部服务。

3. 记录拆分前后差异

重点看：

- 可维护性是否提升
- 异常链路是否可控
- 调试是否变复杂
- 是否真的值得推广到更多模块

### 阶段完成标准

- 至少一个试点模块完成服务化
- 主流程在服务失败时不崩
- 服务接口基本稳定
- 团队能总结出可复用拆分模板

### 这一阶段的产出物

- 天气服务原型
- 论文服务原型
- 主 Agent 调外部服务样例
- 拆分经验总结

---

### Phase 5：实时通话与 Live2D 后端阶段

### 目标

开始实现 [`documents/live2d_webRTC_backend_design.md`](./live2d_webRTC_backend_design.md) 中规划的实时能力。

### 说明

这一阶段建议视为一个独立项目子阶段，不要与前面几个阶段并行大规模展开。

原因很简单：

- 它涉及新协议
- 涉及 WebSocket
- 涉及 WebRTC
- 涉及新的运行时状态机
- 涉及前端联调

这是明显高复杂度模块，应该在前面基础比较稳之后再进入主线。

### 推荐落地顺序

#### 第一步：先做协议和会话骨架

优先实现：

- `CallSession` 数据结构
- 创建会话接口
- 查询状态接口
- 关闭会话接口
- WebSocket 连接生命周期

先不要急着做媒体协商。

#### 第二步：再做 WebSocket 事件链路

先把这些事件跑起来：

- `call.state.changed`
- `assistant.text.start`
- `assistant.text.delta`
- `assistant.text.final`
- `assistant.speaking.start`
- `assistant.speaking.stop`
- `avatar.state`

先用文本驱动，验证前后端事件模型。

#### 第三步：接 WebRTC 信令

再逐步加入：

- offer / answer
- ICE candidate
- peer 生命周期管理

#### 第四步：实现占位视频轨

按设计稿思路，在 WSL 无摄像头环境中优先打通占位视频。

这一步的目标不是“好看”，而是：

- 可测
- 可联调
- 可验证媒体链路

#### 第五步：补充 Live2D 控制事件

在文本和状态事件稳定后，再细化：

- `avatar.expression`
- `avatar.motion`
- `avatar.viseme`

#### 第六步：最后做生产化补强

包括：

- 鉴权
- 超时清理
- 日志监控
- TURN / NAT
- 压测与资源限制

### 阶段完成标准

满足以下条件即可视为实时后端 MVP 成立：

1. 能创建和关闭 1 对 1 通话会话
2. 前端能通过 WebSocket 收到文本和状态事件
3. 前端能通过 WebRTC 收到占位视频轨
4. 前端能收到驱动 Live2D 的结构化事件
5. 不影响现有 `/chat` 文本接口

### 这一阶段的产出物

- 实时通话 API
- WebSocket 事件协议
- WebRTC 占位视频能力
- Live2D 控制事件输出
- 联调说明

## 6. 推荐执行顺序（当前执行策略：实时优先）

结合当前代码状态与团队决策，近期建议按下面顺序执行：

1. 稳住已完成基线（单测、自检、Web MVP 回归脚本）
2. 推进 Phase 5 第四步：接入占位视频/音频轨，先达成可测可联调
3. 补齐实时链路回归与运维文档（会话生命周期、断连清理、错误分支）
4. 收敛 Web 自然语言链路的上游模型超时波动（避免回归脚本不稳定）
5. 在实时 MVP 基本稳定后，再回补服务化试点与更重的产品化细节

这个顺序的核心原因是：

- “独立实时入口 + WS 信令”已经落地，继续推进媒体层的边际收益最高
- 当前改造遵守“只动 transport/skill/bootstrap/main”，主链与 QQ 链路风险可控
- Web MVP 目前已可用，短期以内以稳定维护为主即可
- 先把实时 MVP 闭环做出来，再做更重的产品化和服务化更稳
