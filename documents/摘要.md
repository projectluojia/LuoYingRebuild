# 项目摘要（用于新会话继承记忆）

更新时间：2026-03-22

## 0. 开发原则（当前阶段强约束）

- 不改变主链路（EventHandler/命令分发/现有聊天流程）和 QQ 链路。
- 除 `transport`、`skill`、`bootstrap`、`main` 外，不改其他目录和模块。
- 目标是以最小侵入方式推进新能力，避免引入主链回归风险。

## 1. 当前阶段结论

- 项目已完成 Phase 1 的核心收口：文档、配置说明、基础测试、启动自检已落地。
- Web 端已具备最小可用闭环（身份 + 会话 + 历史 + 聊天）。
- 已完成一轮“仅改边界”的会话策略最小落地：仅改 `transport`、`skill`、`bootstrap`、`main`。
- 实时阶段已完成前两步：独立实时入口 + WS 信令层落地，且 server-side WebRTC 协商链路已补齐关键分支。
- 部署文档与配置说明已补充实时入口启动、健康检查与依赖判定说明。

## 4. 最近一次合并后验收结果

已执行并通过：

- `python -m unittest discover -s tests -v`（32 tests 全通过）
- `cd src && python -m luoying_bot.tools.selfcheck`（PASS）
- `python tests/check_ollama_status.py`（PASS，模型可用）
- 实时入口 smoke：`/realtime/health` + `/realtime/sessions` 可用

当前已识别风险：

- `tests/web_mvp_regression.sh` 在自然语言 `/api/chat` 路径上存在上游模型 `ReadTimeout` 波动（返回 502）
- 同一环境下命令路径（如 `/help`）可返回 200，说明 Web 路由与错误契约本身正常，风险主要在上游模型调用稳定性

本轮已完成的关键修复：

- `/api/chat` 跨用户 session 异常分支已统一转成结构化 HTTP JSON 错误响应（`detail.code/message/status`）
- `/chat` 与 `/api/chat` 的 400/502/500 错误契约已统一
- Web 前端已兼容结构化 `detail`，避免错误提示出现 `[object Object]`
- 位置：
  - [`src/luoying_bot/infra/web/api.py`](../src/luoying_bot/infra/web/api.py)
  - [`src/luoying_bot/infra/web/index.html`](../src/luoying_bot/infra/web/index.html)

## 6. 常用命令（新会话可直接执行）

```bash
# 全量单测
python -m unittest discover -s tests -v

# 实时 transport 回归
python -m unittest tests/test_realtime_ws_transport.py -v

# 启动自检
cd src
python -m luoying_bot.tools.selfcheck

# 启动 Web
cd src
uvicorn luoying_bot.main_web:create_app --factory --host 0.0.0.0 --port 8000

# 启动实时入口
cd src
uvicorn luoying_bot.main_realtime:create_app --factory --host 0.0.0.0 --port 8010

# 实时健康检查
curl -s http://127.0.0.1:8010/realtime/health

# Web MVP 回归（需先启动 Web）
./tests/web_mvp_regression.sh

# Ollama 状态检查
python tests/check_ollama_status.py
```
