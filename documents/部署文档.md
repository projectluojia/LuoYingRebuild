# 部署文档

## 1. 文档目的

本文档用于说明当前仓库在新环境中的部署、启动与排查方式。

注意：

- 本文档基于当前仓库实际代码编写
- 目标是帮助开发者先把环境搭起来
- 本文档同时记录当前已知风险与排查路径

---

## 2. 当前部署结论

当前项目已经具备基础部署结构，但还没有完全达到“开箱即跑”状态。

目前可以明确确认的情况如下：

1. 依赖文件、配置文件、QQ/Web/Realtime 三入口代码都已经存在
2. README 与本文档中的启动命令已统一为先 `cd src` 再启动
3. 历史语法错误与路径问题已修复
4. 当前主要风险转为“依赖缺失、模型接口配置不一致、外部服务可用性（含实时媒体依赖）”

因此，当前阶段更适合把部署理解为：

`先完成环境搭建和启动验证，再聚焦依赖与外部服务配置。`

## 3. 环境要求

建议环境如下：

- Python 3.11+
- Linux 或 Windows 均可
- 能访问所需模型 API
- 若使用 QQ 入口，需要可用的 OneBot WebSocket 服务端

项目依赖位于：

- [`requirements.txt`](../requirements.txt)

当前依赖主要包括：

- `fastapi`
- `uvicorn`
- `websockets`
- `httpx`
- `requests`
- `pydantic`
- `Pillow`
- `python-dotenv`
- `langchain`
- `langchain-openai`
- `langgraph`
- `arxiv`
- `tavily-python`
- `aiortc`（实时协商依赖，包含媒体相关依赖）

## 4. 目录与路径说明

当前仓库中的关键目录如下：

```text
.
├─ .env.example
├─ requirements.txt
├─ documents/
└─ src/
   ├─ data/
   └─ luoying_bot/
```

当前仓库的数据文件实际位于：

- `src/data/userdatabase.json`
- `src/data/reminders.json`
- `src/data/quick_replies.json`
- `src/data/memo/`
- `src/data/scripts/`

而本文档推荐的启动方式是先在仓库根目录执行：

```bash
cd src
```

在这种方式下，配置默认值中的 `./data` 会自然解析到仓库里的 `src/data`。

因此，只要按本文档方式启动，默认数据路径本身就是匹配的。

## 5. 部署步骤

### Step 1：创建虚拟环境

在仓库根目录执行：

```bash
python -m venv .venv
```

激活环境：

Linux / macOS:

```bash
source .venv/bin/activate
```

Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

### Step 2：安装依赖

```bash
pip install -r requirements.txt
```

### Step 3：准备环境变量文件

复制配置模板：

```bash
cp .env.example .env
```

Windows 可手动复制文件或使用：

```powershell
Copy-Item .env.example .env
```

### Step 4：修改 `.env`

最少需要确认以下几类配置：

1. 主模型配置
2. 数据文件路径
3. QQ 入口配置或 Web 入口配置
4. 可选的天气、联网搜索、图片模型配置

如果你按本文档推荐方式 `cd src` 后再启动，那么数据路径保持默认值即可，例如：

```env
DATA_DIR=./data
MEMO_DIR=./data/memo
QUICK_REPLY_FILE=./data/quick_replies.json
USER_DB_FILE=./data/userdatabase.json
REMINDER_DB_FILE=./data/reminders.json
SCRIPT_WORKSPACE_DIR=./data/scripts
```

### Step 5：进入源码目录

当前项目源码位于 `src/` 下，且代码内部导入使用的是 `luoying_bot.xxx`。

因此建议从仓库根目录先进入 `src` 再执行启动命令：

```bash
cd src
```

这样可以直接让 Python 以 `src` 作为当前工作目录解析 `luoying_bot` 包。

## 6. 启动方式

## 6.1 启动 Web API

当前 Web 启动入口文件为：

- [`src/luoying_bot/main_web.py`](../src/luoying_bot/main_web.py)

推荐启动命令：

```bash
cd src
uvicorn luoying_bot.main_web:create_app --factory --host 0.0.0.0 --port 8000
```

说明：

- 不要使用 `uvicorn src.luoying_bot.main_web:create_app`
- 当前代码结构更适合先进入 `src` 再启动

Web 接口当前最小入口为：

- `POST /chat`

## 6.2 启动 QQ 入口

当前 QQ 启动入口文件为：

- [`src/luoying_bot/main_qq.py`](../src/luoying_bot/main_qq.py)

推荐启动命令：

```bash
cd src
python -m luoying_bot.main_qq
```

注意：

- 原 README 中的 `python -m src.luoying_bot.main_qq` 实测会失败
- 直接在仓库根目录 `cd src` 后启动更直接

## 6.3 启动实时入口（WS 信令 + 占位视频轨）

当前实时启动入口文件为：

- [`src/luoying_bot/main_realtime.py`](../src/luoying_bot/main_realtime.py)

推荐启动命令：

```bash
cd src
uvicorn luoying_bot.main_realtime:create_app --factory --host 0.0.0.0 --port 8010
```

建议先检查的接口：

- `GET /realtime/health`
- `POST /realtime/sessions`
- `WS /realtime/ws/{session_id}`

说明：

- 当前实时入口已支持会话、信令（offer/answer/ice）与 server 侧占位视频轨
- 若客户端上行视频轨，服务端会发送 `assistant.text.start/delta/final` 确认事件
- 如果环境未安装 `aiortc`（或 `av`），server-side WebRTC 协商会返回结构化 `signal.error`（`AIORTC_NOT_AVAILABLE`）

## 7. 启动前建议检查项

建议按下面顺序检查：

1. Python 版本是否符合预期

```bash
python --version
```

2. 依赖是否安装完成

```bash
pip list
```

3. `.env` 是否存在

4. `OPENAI_API_KEY` 或本地 Ollama 配置是否已填

5. 数据文件路径是否存在

6. 若走 QQ 链路，`WS_URL` 对应 OneBot 服务是否可连接

7. 若走 Web 链路，是否已经先进入 `src` 目录

8. 若走实时链路，`GET /realtime/health` 是否可用，且 `server_webrtc_enabled` / `aiortc_available` 与预期一致

## 8. 历史问题与当前风险

以下内容用于区分“已修复历史问题”和“当前仍需关注的风险”。

### 8.1 历史问题：README 启动命令曾不正确（已修复）

历史上 README 曾给出：

```bash
python -m src.luoying_bot.main_qq
```

该命令实测报错：

`ModuleNotFoundError: No module named 'luoying_bot'`

原因是：

- 代码内部使用 `from luoying_bot.xxx import ...`
- 并不是 `from src.luoying_bot.xxx import ...`

现已统一改为：

```bash
cd src
python -m luoying_bot.main_qq
```

### 8.2 历史问题：语法错误导致启动失败（已修复）

历史上 `src/luoying_bot/domain/message.py` 存在 f-string 引号错误，已完成修复。

### 8.3 当前风险：依赖缺失会导致模块导入失败

若未安装完整依赖，常见错误为：

- `ModuleNotFoundError: No module named 'httpx'`
- `ModuleNotFoundError: No module named 'fastapi'`
- `ModuleNotFoundError: No module named 'langchain'`

建议优先运行：

```bash
pip install -r requirements.txt
```

### 8.4 当前风险：模型服务接口配置不一致

例如本地 Ollama 若未正确提供 OpenAI 兼容接口，会出现：

- `/v1/chat/completions` 返回 404

建议使用状态检查命令排查：

```bash
python tests/check_ollama_status.py
```

### 8.5 数据路径需要和启动目录一起理解

默认配置使用 `./data`。

如果你按本文档推荐方式先执行：

```bash
cd src
```

那么 `./data` 就会正确指向仓库里的 `src/data`。

真正会出问题的情况是：

- 在仓库根目录直接启动
- 但仍然沿用 `./data` 这组默认值

那样程序才会去找仓库根目录下的 `data/`。

### 8.6 当前风险：实时媒体依赖未安装时，只有信令链路可用

当前 `main_realtime` 已落地会话、信令与占位视频轨，但媒体协商依赖 `aiortc` 与 `av`。

若未安装 `aiortc`（或 `av`），常见表现为：

- 前端向 `to: "server"` 发送 `signal.offer` 后收到 `signal.error`
- `payload.code=AIORTC_NOT_AVAILABLE`

建议排查顺序：

1. 先确认 `GET /realtime/health` 可访问
2. 再确认 `aiortc_available` 字段是否为 `true`
3. 最后再做 WebRTC 协商联调

## 9. Web 模式部署建议

如果当前目标是先验证主逻辑，建议优先使用 Web 模式，而不是先纠缠 QQ 接入。

原因：

1. Web 链路更短，排错更直接
2. 更符合当前阶段“先验证系统能跑”的目标
3. 不依赖 OneBot 环境即可推进大部分后端验证

推荐验证顺序：

1. 配置 `.env`
2. 安装依赖
3. 运行启动自检
4. （可选）运行 Ollama 状态检查
5. 用 Web 模式启动
6. 打开浏览器访问 `/`
7. 先填写身份，再创建会话并发送消息
8. 如需接口级验证，再调用 `POST /chat` 或 `/api/chat`
9. 检查错误分支是否返回结构化 JSON `detail`（`code/message/status`）
10. 运行 Web MVP 回归脚本做稳定性检查
11. 再考虑 QQ 接入问题

---

## 10. QQ 模式部署建议

如果需要验证 QQ 链路，建议额外确认：

1. OneBot 服务端已启动
2. `WS_URL` 对应地址正确
3. 机器人账号已完成平台侧配置
4. `BOT_QQ`、`BOT_NAME` 与实际机器人一致
5. `SPECIFIC_GROUP_IDS` 已正确填写

当前 QQ 模式更适合做：

- transport 调试
- 群聊上下文调试
- 平台特有能力验证

而不适合承担全部主流程验证职责。

## 11. 部署完成后的最小验收

若只看当前阶段，建议以以下标准作为“部署基本成功”：

1. 虚拟环境创建完成
2. 依赖安装完成
3. `.env` 配置完成
4. 模块路径配置正确
5. 启动命令能够进入主程序而不是立刻因环境错误退出
6. 至少能完成一次 Web 或非 QQ 模式下的基本链路验证
7. 若本轮涉及实时链路，至少能完成一次 `POST /realtime/sessions` 与 `GET /realtime/health` 验证

## 12. 启动前自检与基础测试

为了减少“命令一跑就炸，但不知道炸在哪”的情况，仓库里现在提供了两层最基础检查。

### 12.1 启动自检

从仓库根目录执行：

```bash
cd src
python -m luoying_bot.tools.selfcheck
```

这个脚本会检查：

- Python 版本
- 核心依赖是否已安装
- 关键项目模块能否导入
- 数据路径是否存在
- Web 工厂函数能否创建

说明：

- 它不会真的连接 QQ
- 不会真的调用外部模型接口
- 但会明确告诉你当前卡在“缺依赖”还是“模块导入失败”

### 12.2 Smoke Tests

从仓库根目录执行：

```bash
python -m unittest discover -s tests -v
```

当前这组测试主要覆盖：

- `UniMessage`
- `QuickReplyService`
- `ScriptWorkspaceService`
- `EventHandler`
- `CommandDispatcher`
- `ReminderService`
- `Web API` 错误契约（`tests/test_web_api_regression.py`）

适用场景：

- 改完基础代码后快速回归
- 部署前确认最底层本地能力没有明显坏掉

### 12.3 Web MVP 回归脚本（Ubuntu 可用）

如果你已经启动了 Web 服务（`http://127.0.0.1:8000`），可以运行：

```bash
./tests/web_mvp_regression.sh
```

也可指定地址：

```bash
./tests/web_mvp_regression.sh http://127.0.0.1:8000
```

该脚本会覆盖：

- Web 首页可达性检查
- `POST /api/sessions` 创建会话
- `GET /api/sessions` 会话列表查询
- `POST /api/chat` 发送消息并拿到回复
- `GET /api/sessions/{id}/messages` 历史消息查询
- 跨用户访问错误分支是否返回结构化 JSON `detail`（`code/message/status`）

适用场景：

- Web MVP 发布前快速回归
- 定位“前端 JSON 解析报错”类问题
- 验证会话主链路稳定性

### 12.4 Ollama 状态检查

如果你在使用本地 Ollama，建议在聊天前先检查服务状态。

从仓库根目录执行：

```bash
python tests/check_ollama_status.py
```

这个脚本会尽可能覆盖常见错误状态：

- `OLLAMA_BASE_URL` 非法
- Ollama 服务连接失败 / 超时
- `/api/version` 或 `/api/tags` 返回 HTTP 错误
- 响应不是合法 JSON
- `models` 字段缺失
- `models` 列表为空（按错误处理）

判定规则：

- 在线模型列表非空 => 通过
- 在线模型列表为空 => 失败

可选严格模式（要求配置模型必须在线）：

```bash
python tests/check_ollama_status.py --strict-model-match
```

### 12.5 Realtime 入口最小验收

如果你已经启动了实时服务（`http://127.0.0.1:8010`），可先做最小 HTTP 验证。

健康检查：

```bash
curl -s http://127.0.0.1:8010/realtime/health
```

创建会话：

```bash
curl -s -X POST http://127.0.0.1:8010/realtime/sessions \
  -H "Content-Type: application/json" \
  -d '{"owner_id":"u_test","metadata":{"source":"manual_check"}}'
```

验证重点：

- `status` 为 `ok`
- 会话返回 `session_id`
- `aiortc_available` 字段与你当前环境一致

可进一步执行端到端协商 smoke（仓库根目录）：

```bash
python tests/realtime_offer_smoke.py --base-url http://127.0.0.1:8010
```

若要验证“客户端上行视频后，服务端返回 `assistant.text.final` 确认”：

```bash
python tests/realtime_offer_smoke.py --base-url http://127.0.0.1:8010 --send-video
```

若要强校验“语义分析文本事件”：

```bash
python tests/realtime_offer_smoke.py --base-url http://127.0.0.1:8010 --require-semantic --recv-timeout 45
```
