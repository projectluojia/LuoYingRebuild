# 部署文档

## 1. 文档目的

本文档用于说明当前仓库在新环境中的部署、启动与排查方式。

注意：

- 本文档基于当前仓库实际代码编写
- 目标是帮助开发者先把环境搭起来
- 本文档同时记录当前已发现的启动阻塞项

---

## 2. 当前部署结论

当前项目已经具备基础部署结构，但还没有完全达到“开箱即跑”状态。

目前可以明确确认的情况如下：

1. 依赖文件、配置文件、QQ/Web 双入口代码都已经存在
2. 启动方式需要显式处理 Python 模块路径
3. README 中当前给出的 QQ 启动命令不正确
4. 当前代码中存在至少一个会直接阻塞启动的语法错误

因此，当前阶段更适合把部署理解为：

`先完成环境搭建和启动验证，再逐步消除代码级阻塞项。`

## 4. 目录与路径说明

当前仓库中的关键目录如下：

```text
.
├─ .env.example
├─ requirements.txt
├─ documents/
└─ src/
   ├─ data/
   └─ luoying_bot/
```

当前仓库的数据文件实际位于：

- `src/data/userdatabase.json`
- `src/data/reminders.json`
- `src/data/quick_replies.json`
- `src/data/memo/`
- `src/data/scripts/`

而本文档推荐的启动方式是先在仓库根目录执行：

```bash
cd src
```

在这种方式下，配置默认值中的 `./data` 会自然解析到仓库里的 `src/data`。

因此，只要按本文档方式启动，默认数据路径本身就是匹配的。

## 5. 部署步骤

### Step 1：创建虚拟环境

在仓库根目录执行：

```bash
python -m venv .venv
```

激活环境：

Linux / macOS:

```bash
source .venv/bin/activate
```

Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

### Step 2：安装依赖

```bash
pip install -r requirements.txt
```

### Step 3：准备环境变量文件

复制配置模板：

```bash
cp .env.example .env
```

Windows 可手动复制文件或使用：

```powershell
Copy-Item .env.example .env
```

### Step 4：修改 `.env`

最少需要确认以下几类配置：

1. 主模型配置
2. 数据文件路径
3. QQ 入口配置或 Web 入口配置
4. 可选的天气、联网搜索、图片模型配置

如果你按本文档推荐方式 `cd src` 后再启动，那么数据路径保持默认值即可，例如：

```env
DATA_DIR=./data
MEMO_DIR=./data/memo
QUICK_REPLY_FILE=./data/quick_replies.json
USER_DB_FILE=./data/userdatabase.json
REMINDER_DB_FILE=./data/reminders.json
SCRIPT_WORKSPACE_DIR=./data/scripts
```

### Step 5：进入源码目录

当前项目源码位于 `src/` 下，且代码内部导入使用的是 `luoying_bot.xxx`。

因此建议从仓库根目录先进入 `src` 再执行启动命令：

```bash
cd src
```

这样可以直接让 Python 以 `src` 作为当前工作目录解析 `luoying_bot` 包。

## 6. 启动方式

## 6.1 启动 Web API

当前 Web 启动入口文件为：

- [`src/luoying_bot/main_web.py`](../src/luoying_bot/main_web.py)

推荐启动命令：

```bash
cd src
uvicorn luoying_bot.main_web:create_app --factory --host 0.0.0.0 --port 8000
```

说明：

- 不要使用 `uvicorn src.luoying_bot.main_web:create_app`
- 当前代码结构更适合先进入 `src` 再启动

Web 接口当前最小入口为：

- `POST /chat`

## 6.2 启动 QQ 入口

当前 QQ 启动入口文件为：

- [`src/luoying_bot/main_qq.py`](../src/luoying_bot/main_qq.py)

推荐启动命令：

```bash
cd src
python -m luoying_bot.main_qq
```

注意：

- 原 README 中的 `python -m src.luoying_bot.main_qq` 实测会失败
- 直接在仓库根目录 `cd src` 后启动更直接

---

## 7. 启动前建议检查项

建议按下面顺序检查：

1. Python 版本是否符合预期

```bash
python --version
```

2. 依赖是否安装完成

```bash
pip list
```

3. `.env` 是否存在

4. `OPENAI_API_KEY` 或本地 Ollama 配置是否已填

5. 数据文件路径是否存在

6. 若走 QQ 链路，`WS_URL` 对应 OneBot 服务是否可连接

7. 若走 Web 链路，是否已经先进入 `src` 目录

## 8. 当前已知阻塞项

以下问题是本次核对部署流程时已经实际验证到的。

### 8.1 原 README 中的 QQ 启动命令不正确

README 当前给出：

```bash
python -m src.luoying_bot.main_qq
```

该命令实测报错：

`ModuleNotFoundError: No module named 'luoying_bot'`

原因是：

- 代码内部使用 `from luoying_bot.xxx import ...`
- 并不是 `from src.luoying_bot.xxx import ...`

因此应改为：

```bash
cd src
python -m luoying_bot.main_qq
```

### 8.2 当前代码存在语法错误，会阻塞启动

### 8.3 数据路径需要和启动目录一起理解

默认配置使用 `./data`。

如果你按本文档推荐方式先执行：

```bash
cd src
```

那么 `./data` 就会正确指向仓库里的 `src/data`。

真正会出问题的情况是：

- 在仓库根目录直接启动
- 但仍然沿用 `./data` 这组默认值

那样程序才会去找仓库根目录下的 `data/`。

## 9. Web 模式部署建议

如果当前目标是先验证主逻辑，建议优先使用 Web 模式，而不是先纠缠 QQ 接入。

原因：

1. Web 链路更短，排错更直接
2. 更符合当前阶段“先验证系统能跑”的目标
3. 不依赖 OneBot 环境即可推进大部分后端验证

推荐验证顺序：

1. 修复当前语法错误
2. 配置 `.env`
3. 用 Web 模式启动
4. 调用 `POST /chat` 做最小链路验证
5. 再考虑 QQ 接入问题

---

## 10. QQ 模式部署建议

如果需要验证 QQ 链路，建议额外确认：

1. OneBot 服务端已启动
2. `WS_URL` 对应地址正确
3. 机器人账号已完成平台侧配置
4. `BOT_QQ`、`BOT_NAME` 与实际机器人一致
5. `SPECIFIC_GROUP_IDS` 已正确填写

当前 QQ 模式更适合做：

- transport 调试
- 群聊上下文调试
- 平台特有能力验证

而不适合承担全部主流程验证职责。

---

## 11. 部署完成后的最小验收

若只看当前阶段，建议以以下标准作为“部署基本成功”：

1. 虚拟环境创建完成
2. 依赖安装完成
3. `.env` 配置完成
4. 模块路径配置正确
5. 启动命令能够进入主程序而不是立刻因环境错误退出
6. 至少能完成一次 Web 或非 QQ 模式下的基本链路验证
