# 配置说明

## 1. 文档目的

本文档用于解释当前项目的环境变量配置项，以及这些配置在实际代码中的作用范围。

配置读取入口位于：

- [`src/luoying_bot/config.py`](../src/luoying_bot/config.py)

配置模板位于：

- [`.env.example`](../.env.example)

---

## 2. 使用方式

项目通过 `python-dotenv` 自动读取仓库根目录下的 `.env` 文件。

建议做法：

1. 复制 `.env.example` 为 `.env`
2. 按实际运行模式修改配置
3. 启动前确认关键项已填写

说明：

- 本文档主要覆盖“环境变量配置”本身。
- Web 错误响应结构（如 `detail.code/message/status`）属于接口契约，不通过 `.env` 配置切换。

---

## 3. 最小必填配置

如果你只是想先把项目跑起来，至少需要关注下面这些配置。

### 3.1 主模型配置

```env
OPENAI_BASE_URL=
OPENAI_API_KEY=
OPENAI_MODEL=
```

作用：

- 驱动主 Agent
- Web / QQ 两条主链路都会用到

如果未开启本地 Ollama，这一组基本属于核心必填项。

### 3.2 数据路径配置

```env
DATA_DIR=
MEMO_DIR=
QUICK_REPLY_FILE=
USER_DB_FILE=
REMINDER_DB_FILE=
SCRIPT_WORKSPACE_DIR=
```

作用：

- 决定用户数据、提醒、快捷回复、备忘录、脚本工作区的存放位置

如果你按当前推荐方式先 `cd src` 再启动，建议保持默认值：

```env
DATA_DIR=./data
MEMO_DIR=./data/memo
QUICK_REPLY_FILE=./data/quick_replies.json
USER_DB_FILE=./data/userdatabase.json
REMINDER_DB_FILE=./data/reminders.json
SCRIPT_WORKSPACE_DIR=./data/scripts
```

### 3.3 Web 或 QQ 模式相关配置

如果走 Web 模式：

- 主模型配置可用即可

如果走 QQ 模式，还需要：

```env
WS_URL=
BOT_QQ=
BOT_NAME=
SPECIFIC_GROUP_IDS=
```

## 4.5 本地 Ollama 配置

```env
USE_LOCAL_OLLAMA=false
OLLAMA_BASE_URL=http://127.0.0.1:11434/v1
OLLAMA_API_KEY=ollama
OLLAMA_MAIN_MODEL=qwen2.5:7b-instruct
OLLAMA_CODING_MODEL=qwen2.5-coder:7b
OLLAMA_IMAGE_MODEL=llava:7b
```

> **重要提醒：不要更改 ollama 模型存储位置。**

作用：

- 统一切换主 Agent、编程子 Agent、图片子 Agent 到本地 Ollama

影响范围：

- [`src/luoying_bot/bootstrap.py`](../src/luoying_bot/bootstrap.py)
- [`src/luoying_bot/bootstrap_web.py`](../src/luoying_bot/bootstrap_web.py)
- [`src/luoying_bot/application/agent/skills/coding_agent_skill.py`](../src/luoying_bot/application/agent/skills/coding_agent_skill.py)
- [`src/luoying_bot/application/agent/skills/image_agent_skill.py`](../src/luoying_bot/application/agent/skills/image_agent_skill.py)

### `USE_LOCAL_OLLAMA`

核心开关。

当为 `true` 时：

- 主 Agent 不再使用 `OPENAI_*`
- 编程子 Agent 不再使用 `CODER_*`
- 图片子 Agent 不再使用 `IMAGE_*`

而改为统一使用 `OLLAMA_*`

### `OLLAMA_BASE_URL`

本地 Ollama 的 OpenAI 兼容接口地址。

### `OLLAMA_API_KEY`

本地 Ollama 默认通常可写成 `ollama`。

### `OLLAMA_MAIN_MODEL`

主 Agent 使用的本地模型。

### `OLLAMA_CODING_MODEL`

编程子 Agent 使用的本地模型。

### `OLLAMA_IMAGE_MODEL`

图片子 Agent 使用的本地模型。

### 配置后建议检查

配置好 Ollama 相关变量后，建议先执行状态检查命令：

```bash
python tests/check_ollama_status.py
```

若要把“配置模型不在线”也判定为失败，可用严格模式：

```bash
python tests/check_ollama_status.py --strict-model-match
```

## 4.7 数据路径配置

```env
DATA_DIR=./data
MEMO_DIR=./data/memo
QUICK_REPLY_FILE=./data/quick_replies.json
USER_DB_FILE=./data/userdatabase.json
REMINDER_DB_FILE=./data/reminders.json
SCRIPT_WORKSPACE_DIR=./data/scripts
```

作用：

- 决定项目运行时数据位置

影响范围：

- 用户资料
- 提醒事项
- 快捷回复
- 备忘录
- 用户脚本工作区

### 重要说明

当前仓库实际已有数据位于：

- `src/data`

而项目推荐启动方式是先进入 `src` 目录再启动。

因此在实际运行时，默认配置中的：

- `./data`

会自然对应到仓库里的：

- `src/data`

所以在推荐启动方式下，默认值就是合理的，一般不需要额外改写数据路径。

### `DATA_DIR`

当前代码中这个字段更多像总目录保留项，本身未被各服务直接统一引用。

这意味着：

- 它不会自动推导其他路径
- 仍需要分别配置各子路径

### `MEMO_DIR`

备忘录目录。

### `QUICK_REPLY_FILE`

快捷回复 JSON 文件路径。

### `USER_DB_FILE`

用户资料数据库路径。

### `REMINDER_DB_FILE`

提醒数据库路径。

### `SCRIPT_WORKSPACE_DIR`

用户脚本工作区根目录。

## 4.9 权限 / 群组 / 触发前缀

```env
OPS=2564664062
SPECIFIC_GROUP_IDS=1087614373,955571307
TRIGGER_PREFIX=/,!,。
```

作用：

- 管理员权限
- 群启用范围
- 触发前缀配置

影响范围：

- [`src/luoying_bot/bootstrap.py`](../src/luoying_bot/bootstrap.py)
- [`src/luoying_bot/bootstrap_web.py`](../src/luoying_bot/bootstrap_web.py)
- [`src/luoying_bot/application/event_handler.py`](../src/luoying_bot/application/event_handler.py)
- [`src/luoying_bot/application/commands/builtin.py`](../src/luoying_bot/application/commands/builtin.py)

字段说明：

### `OPS`

管理员账号列表，使用逗号分隔。

### `SPECIFIC_GROUP_IDS`

允许启用机器人的群列表。

注意：

- QQ 模式下，未启用群会直接沉默
- Web 模式一般不受这个配置影响

### `TRIGGER_PREFIX`

触发前缀列表，使用逗号分隔。

注意：

- 当前代码会读取这个列表
- 但 `EventHandler` 中并未严格依赖此前缀来放行普通消息
- 命令识别的核心仍然是消息是否以 `/` 开头

因此它当前更像预留配置，而不是强控制项。

## 6. 当前配置层面的注意事项

### 6.1 `.env.example` 与代码默认值并不完全一致

例如：

- `WEATHER_BASE_URL` 在模板中为空
- 但在代码中有默认值

因此配置判断应以 `config.py` 为准，而不是只看模板。

### 6.2 有些配置项当前是保留项或弱生效项

例如：

- `CODER_TEMPERATURE`
- `DATA_DIR`
- `SCRIPT_SEND_CHUNK_SIZE`
- `TRIGGER_PREFIX`

这些字段不是没意义，但当前代码并没有完全把它们用成强控制项。

### 6.3 数据路径和启动目录必须配套理解

这是当前最容易踩坑的配置问题。

### 6.3 Web 错误契约不是环境变量项

当前 `/chat`、`/api/chat` 的错误返回已统一为结构化 JSON `detail`：

```json
{
  "detail": {
    "code": "BAD_REQUEST",
    "message": "xxx",
    "status": 400
  }
}
```

这部分不通过 `.env` 控制，建议通过回归测试保障：

- [`tests/test_web_api_regression.py`](../tests/test_web_api_regression.py)

如果不注意这一点：

- 在仓库根目录启动时，`./data` 会指向根目录下的 `data`
- 在 `src` 目录启动时，`./data` 才会指向仓库中的 `src/data`

所以关键不是“必须改配置值”，而是“启动目录和路径配置要一致”。

## 7. 推荐的当前 `.env` 基线

如果你的目标是先在当前仓库里把 Web 调试跑起来，建议以这个方向为基线：


如果后续再接 QQ，再继续补：

```env

```

## 8. 一句话总结

当前配置系统已经基本够用，但还没有完全收口到“配置项、模板、README、目录结构完全一致”的状态。

现阶段最重要的配置动作只有两件事：

1. 先把模型配置配对
2. 让启动目录与 `./data` 这组默认值保持一致

这样能避免大多数初期启动问题。
