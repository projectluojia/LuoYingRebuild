# 珞樱 Luoying

珞樱（Luoying）是一个面向校园场景构建的多能力 Agent 系统。它并不把自己定位成“某个平台上的聊天机器人”，而是把自己定位成一个**可被多端接入、可持续扩展的 Agent 内核**。

当前项目最成熟的入口是 QQ（OneBot WebSocket），同时已经具备 Web 入口雏形。项目的长期目标不是继续把单个平台做得越来越重，而是把已经验证过的能力收口成统一抽象，在此基础上建设新的 Web 主链路，并逐步支持流式输出、文件上传、多模态输入输出、图片生成以及实时交互能力。

---

## 1. 项目定位

珞樱要解决的问题不是“自动回复”，而是：

- 如何把来自不同平台的输入统一收口
- 如何把命令、自然语言请求、任务型请求放进同一条处理链路
- 如何在群聊 / 私聊 / Web 会话中维护清晰的上下文
- 如何让稳定能力下沉为服务，而不是把所有事情都压给大模型
- 如何在后续接入 Web、多模态、实时交互时，尽量不推翻已有架构

因此，珞樱更适合被理解为：

- 一个 Agent 后端骨架
- 一个校园场景的能力编排系统
- 一个可以继续接入 QQ / Web / 其他平台的多端内核

---

## 2. 当前状态概览

### 已经比较成熟的部分

- QQ OneBot WebSocket 接入链路
- 统一消息对象 `UniMessage`
- 统一上下文对象 `ChatContext`
- 命令系统（`CommandDispatcher`）
- 主 Agent + Skill 体系
- 业务服务层（提醒、备忘录、用户信息、长期记忆、脚本工作区等）
- 消息调度器（跨会话并发、会话内串行）
- JSON / 文本持久化
- 计划任务与恢复机制

### 目前仍处于早期或待重构的部分

- Web 入口仍然较薄，当前以简单文本聊天接口为主
- Web 尚未形成独立 transport 装配
- 系统输出模型仍偏“最终文本回复”，不利于流式、多模态与生成式能力扩展
- 文件上传、图片上传、结构化事件返回仍需系统化建设
- 图片生成、实时音视频仍处于下一阶段规划中

## 3. 架构设计原则

### 3.1 平台无关优先

平台接入层只负责：

- 接收事件
- 解析消息
- 调用统一处理链路
- 把输出发送回平台

真正的业务逻辑、能力编排、记忆与服务，不应写死在某个平台实现中。

### 3.2 Command / Agent / Service 分工

不是所有问题都交给大模型。

- **Command**：高确定性、规则清晰、参数明确的操作
- **Agent**：多步思考、技能选择、复杂自然语言任务
- **Service**：稳定、可复用、平台无关的底层能力

这种混合架构比“纯规则”或“纯大模型”都更稳。

### 3.3 统一消息模型

不同平台来的消息最终都收敛到统一对象：

- `UniMessage`
- `MessageSegment`
- `ChatContext`

这让平台差异尽量停留在边界层，而不是渗透进整个业务系统。

### 3.4 会话内串行、跨会话并发

`MessageProcessor` 负责确保：

- 不同会话可以并发处理
- 同一会话内部保持串行

这对多用户场景、记忆一致性、后续流式响应和实时交互都很重要。

## 5. 核心处理链路

### 5.1 QQ 链路

```text
QQ / OneBot 事件
  -> QQWsTransport
  -> UniMessage
  -> MessageProcessor
  -> EventHandler
  -> Command / Agent / Service
  -> ChatTransport.send_text(...)
```

### 5.2 当前 Web 链路

```text
HTTP /chat
  -> WebApiFactory
  -> UniMessage.from_web_text(...)
  -> EventHandler.handle(...)
  -> 返回最终文本
```

这条 Web 链路目前可以用于基础文本调试，但还不是完整的产品级主链路。下一阶段会进一步重构为独立的 Web transport / session 模型，并接入统一消息处理流程。

## 6. 主要模块说明

### 6.1 `ChatTransport`

`ChatTransport` 是平台接入的抽象边界，定义了：

- 收消息 `recv_message`
- 发文本 `send_text`
- mention 格式化 `format_mention`
- 获取群成员
- 拉取消息
- 下载图片
- 下载文件（预留）
- 上传文件

当前最完整的实现是 `QQWsTransport`。

后续 Web、多模态与实时能力建设，会继续围绕 transport 边界扩展。

### 6.2 `EventHandler`

统一业务入口，负责：

- 运行时状态判断
- 快捷回复
- 命令派发
- Agent 调用
- 风控输入输出处理

它的价值在于把平台事件转换成统一业务流程，而不是在各个平台入口里重复写判断逻辑。

### 6.3 `MessageProcessor`

负责并发调度：

- 同一会话串行，避免记忆乱序
- 多个会话并发，提升整体吞吐

这是后续做 Web、多用户和流式响应时的重要基础设施。

### 6.4 `AgentService`

主 Agent 使用受控 ReAct 风格循环：

- 读取系统提示词与历史上下文
- 获取可用技能摘要
- 让模型只输出 JSON 行动或最终答案
- 控制最大步骤数、技能超时和总预算

这使得 Agent 既有一定灵活性，又不会完全失控。

### 6.5 Skill 系统

当前项目中已经存在多类技能，例如：

- 提醒事项
- 天气查询
- 联网搜索
- 备忘录
- 群信息（QQ）
- 图片理解
- 编程子 Agent
- 用户长期记忆

Skill 的目标不是堆功能，而是把“可被 Agent 选择的能力”封装成统一接口。

## 8. 下一阶段路线图

### Phase 1：抽象收口

目标：明确平台相关与平台无关边界。

重点：

- 进一步收敛 transport 能力边界
- 提炼统一输入模型
- 提炼统一输出事件模型
- 清理 QQ 假设对核心层的渗透

### Phase 2：Web 主链路建设

目标：让 Web 从“调试入口”升级为“正式入口”。

重点：

- 独立 Web transport / session 模型
- 统一接入 `MessageProcessor`
- SSE 或 WebSocket 流式输出
- 文件上传与附件管理
- 图片上传与多模态会话支持
- 前后端协议结构化

### Phase 3：生成式多模态

目标：让系统不仅能理解图片，也能产出图片等媒体结果。

重点：

- 图片生成 skill / service
- 非文本结果的前端展示协议
- 任务状态回传
- 文件与媒体产物管理

### Phase 4：实时交互能力

目标：为视频通话、实时语音和更复杂的互动模式预留架构。

重点：

- 长连接
- 房间 / 会话管理
- 实时事件流
- 音视频信令与状态同步

## 9. 为什么接下来不继续以 QQ 为重心

QQ 并不是被放弃，而是已经完成了验证任务。

当前阶段继续把精力主要投入在 QQ 平台细节上，会有三个问题：

1. 平台细节继续侵入核心逻辑
2. 真正重要的新能力很难自然落地
3. 架构会越来越偏向单平台，而不是平台内核

因此，下一阶段更合理的方向是：

- 保持 QQ 链路可用
- 但把新增建设重点转向抽象层和 Web 主链路

这不是“从 QQ 转去做 Web 页面”，而是“把 QQ 已经验证出来的内核能力产品化”。

## 10. 运行说明

### 10.1 环境准备

建议使用 Python 3.11+。

安装依赖：

```bash
pip install -r requirements.txt
```

复制环境变量模板：

```bash
cp .env.example .env
```

根据需要配置：

- QQ WebSocket 地址与鉴权
- LLM 供应商与模型
- 天气 API
- 搜索 API
- 脚本工作区与数据目录

### 10.2 启动 QQ 入口

```bash
python -m src.luoying_bot.main_qq
```

或按你的项目实际运行方式启动。

### 10.3 启动 Web 入口

当前 Web 入口仍较基础，通常需要结合你自己的 ASGI 启动方式，例如：

```bash
uvicorn luoying_bot.main_web:create_app --factory --host 0.0.0.0 --port 8000
```

请注意：当前 Web 入口主要用于基础文本聊天调试，后续会继续演进。

## 11. 已知现状与后续重构方向

### 当前 Web 入口的问题

- 只支持纯文本输入
- 返回的是整段文本，不支持流式
- 还没有文件上传 / 图片上传协议
- 当前装配仍复用了 QQ 容器思路
- 没有完整的前端事件模型

### 当前响应模型的问题

系统目前主要围绕“最终返回一段文本”来设计。

这对于 QQ 文本聊天够用，但不利于：

- 流式输出
- 图片生成
- 文件返回
- 任务状态反馈
- 实时音视频控制

后续会逐步升级为统一输出事件模型。

## 12. 开发建议

如果你接下来要继续参与这个项目，建议优先遵守以下原则：

1. 不要把平台细节直接写进核心业务层
2. 新能力尽量先思考它应该是 command、skill 还是 service
3. 优先复用 `UniMessage` / `ChatContext` / `MessageProcessor` 这些统一机制
4. 新的 Web 能力不要只做“能跑”，要思考后续是否能承接多模态和实时交互
5. README、接口协议与代码应同步更新，避免“架构已经变了，但文档还停在旧阶段”

## 14. 当前结论

珞樱的前一阶段，重点是“把 Agent 在 QQ 上真正跑起来”；

珞樱的下一阶段，重点是“把已经验证出来的能力收口成统一内核，并以 Web 主链路为突破口，进入多端、多模态、实时交互的阶段”。

这也是当前项目最值得投入的方向。
